# Crawl Output Observability

Date: 2026-03-27
Status: validated design
Topic: improve crawler output messages for pipeline monitoring and troubleshooting

## Context

The crawler currently logs from inside individual fetch result handlers in `src/crawl.ts`.
That worked while the flow was mostly linear, but with parallel crawling it now produces
interleaved and misleading output:

- best-effort markdown guesses log as independent red errors even when a sibling guess later succeeds
- related messages from the same original URL are split apart in the terminal
- the logs do not clearly distinguish between:
  - expected fallback misses
  - retryable blockers such as rate limits
  - terminal failures that actually make the run worse
- CI output is harder to trust because the loudest lines are not always the most important ones

The user goal is not only "more logs". The goal is:

1. more transparent process reporting
2. fewer false-positive errors
3. grouped output for guessed URLs
4. enough detail to debug red pipelines without re-running locally

## Assumptions

- The crawler remains a plain Node CLI using `console.*`; no external logging library is introduced.
- The crawl behavior does not change. This is an observability redesign, not a fetch policy redesign.
- Human-readable pipeline logs are the priority. Machine-readable JSON logs are out of scope for this pass.
- Best-effort guesses are expected behavior and should only become error-level when the whole guess group fails.
- The implementation can add local helper types/functions and small state objects in `src/crawl.ts`.

## Approaches Considered

### Option A: minimal suppression patch

Change only the existing error prints:

- suppress red errors for `bestEffort` URLs
- print one final error when all guesses fail

Pros:

- smallest code change
- directly fixes the loudest false positives

Cons:

- still leaves the rest of the crawl output ad hoc
- does not improve process visibility for rate limits, retries, or blocked progress
- still risks messy interleaving because there is no explicit grouped event model

### Option B: structured plain-text lifecycle logging with grouped guess outcomes

Introduce a small internal logging model while keeping console output:

- stable severity prefixes
- one logical log event per terminal decision
- grouped buffering for guess families
- more explicit process, retry, block, and summary messages

Pros:

- fixes the guess-noise problem at the right abstraction level
- improves traceability across the whole crawl
- keeps scope local and compatible with existing CLI usage
- avoids overbuilding

Cons:

- requires a small amount of new state in the crawl loop
- test coverage needs to expand around emitted messages

### Option C: full structured event stream / JSON logger

Build a generic event emitter with optional JSON output, correlation ids, and richer sinks.

Pros:

- strongest long-term observability foundation
- easiest to feed into external log processors later

Cons:

- too wide for the current problem
- larger refactor than needed
- adds design surface the project does not yet need

## Recommendation

Choose Option B.

It solves the real problem instead of only hiding symptoms, but it stays narrow:

- no new dependency
- no new telemetry backend
- no new crawl semantics

The implementation should add just enough structure to make log output trustworthy under
parallel execution and useful in CI investigations.

## Section 1: Log Model

### Design

Use a small internal formatter that emits stable, human-readable plain-text events.
Each logical event should be emitted with a single `console.log`, `console.warn`, or
`console.error` call so related lines stay together even when fetches resolve in parallel.

Recommended severity model:

- `INFO`: normal crawl lifecycle and progress
- `NOTICE`: expected fallback behavior that is useful for traceability but should not attract attention
- `WARN`: retrying or temporarily blocked work
- `ERROR`: terminal failure for a primary URL, terminal failure for a whole guess group, or run-level abort
- `SUMMARY`: end-of-seed and end-of-run condensed reports

Recommended output shape:

```text
[INFO] seed.start seed=https://developers.openai.com/api/docs/llms.txt scope=https://developers.openai.com/api/docs/ concurrency=10
[WARN] fetch.rate_limited url=https://example.com/docs/page retry_in_ms=5000 domain=example.com action=pause-domain,requeue-primary consecutive_429s=2
[NOTICE] fetch.guess_resolved original=https://example.com/guides/text/ winner=https://example.com/guides/text.md attempts=3
  - original  https://example.com/guides/text/           -> 404
  - guess     https://example.com/guides/text.md         -> success saved_as=example.com/guides/text.md
  - guess     https://example.com/guides/text/index.md   -> 404
[ERROR] fetch.guess_failed original=https://example.com/guides/text/ attempts=3
  - original  https://example.com/guides/text/           -> 404
  - guess     https://example.com/guides/text.md         -> 404
  - guess     https://example.com/guides/text/index.md   -> 404
```

The exact wording can change, but the structure should stay stable.

### Why this level of structure

- CI users can scan severity prefixes quickly.
- Key-value style details are easy to grep.
- A single call per logical event avoids the current multi-line interleaving problem.
- This does not require a generic logger package.

### Autonomous review

This stays within scope. It improves readability and traceability without introducing a
configuration surface or a large abstraction layer.

## Section 2: Guess Grouping

### Design

Introduce an explicit "guess group" for each original URL that spawns best-effort guesses.

Each group tracks:

- `originalUrl`
- ordered candidate list
- whether each candidate is `original` or `guess`
- terminal outcome for each candidate
- winner, if any
- whether the group has already been emitted

The original URL is the identity of the group. All guessed variants belong to that group.

Recommended behavior:

1. When a URL generates markdown guesses, create or reuse a group rooted at the original URL.
2. Do not log candidate failures immediately.
3. Store each candidate result in group state as it resolves.
4. If any candidate succeeds:
   - mark the group resolved
   - emit one `NOTICE` or low-noise `INFO` block for the original URL
   - include all resolved attempts so far, highlighting the winner
   - suppress standalone candidate errors entirely
5. If all candidates fail:
   - emit one `ERROR` block for the original URL
   - include the outcome of every attempted candidate
6. Emit the group block once, as one atomic console call.

This directly matches the user requirement:

- grouped URLs may execute independently
- grouped URLs must report together
- only the original URL should own the visible message

### Important semantic rule

Best-effort candidate failures are not first-class errors on their own.

They are detail lines inside a grouped resolution event. They only become red output when
the whole group fails.

### Edge cases

- If the original URL itself succeeds, no guess group summary is needed.
- If the original URL fails but a guess succeeds, emit a single grouped success/notice block.
- If one guess succeeds early and another guess finishes later, the later result should not emit
  another standalone log line. The group should already be closed.
- If implementation cannot cancel sibling in-flight guesses, that is acceptable. The logging
  contract is still one grouped message only.

### Autonomous review

This section resolves the actual false-positive problem and is consistent with the current
parallel queue architecture, where sibling guesses may finish in any order.

## Section 3: Process and Blocking Messages

### Design

Make the crawl output narrate the lifecycle clearly enough for CI:

#### Run start

Emit one start block with:

- content directory
- downloads directory
- concurrency per domain
- seed count
- seed list summary

#### Seed start

Emit one line per seed with:

- seed URL
- primary scope
- additional scope prefixes
- local prefix

#### Save result

Keep save messages, but make them informative:

- include save status: `new`, `changed`, `unchanged`
- include source URL and saved path when practical

Example:

```text
[INFO] content.saved status=changed url=https://raw.githubusercontent.com/org/repo/main/README.md path=openai/raw.githubusercontent.com/org/repo/main/README.md
```

#### Temporary blockers

When a rate limit or retry happens, report:

- affected URL
- domain
- retry delay
- whether the URL is requeued or failed
- current consecutive retry count if relevant
- what is blocking forward progress

Example:

```text
[WARN] fetch.rate_limited url=https://example.com/docs/a retry_in_ms=5000 domain=example.com action=pause-domain,requeue-primary consecutive_429s=1
```

This tells the pipeline operator that the run is waiting, not hung.

#### Terminal blockers

When the crawler gives up or aborts, report:

- URL or seed involved
- terminal reason
- retry history / consecutive count
- next effect on the run

Examples:

- primary URL exhausted retries
- three consecutive 429 responses triggered abort
- unexpected processing exception escaped normal handling

#### Seed summary

Emit one end-of-seed summary with:

- fetched count
- success counts by reason
- skipped counts by reason
- failed count
- whether the seed aborted
- notable blockers encountered

#### Run summary

Emit one final summary with:

- total fetched pages
- metadata path
- whether result is `success`, `partial`, or `aborted`
- total failures
- total rewritten markdown files
- seeds completed vs aborted

### Non-goal within this section

Do not log every queue submission or every in-flight transition. That would create high-volume
noise without materially improving diagnosis for this crawler size.

### Autonomous review

This is the correct verbosity level for CI: higher signal around lifecycle and blockers, but no
per-request spam for normal control flow.

## Section 4: Error and Severity Policy

### Design

Adopt explicit severity rules so red lines actually mean something.

#### `ERROR`

Use only for:

- full guess group failure
- primary URL terminal failure after retry policy is exhausted
- unexpected processing exception
- crawler abort condition

#### `WARN`

Use for:

- rate limiting
- retrying a primary URL
- degraded but still progressing situations

#### `NOTICE`

Use for:

- successful fallback from original URL to guessed markdown URL
- optional low-noise skip summaries that are expected in normal crawling

#### `INFO`

Use for:

- start/end summaries
- content save events
- link rewrite summary
- out-of-scope or non-text skips when they are not abnormal for the current path

### Specific policy change requested by the user

404 and 406 on guessed URLs should never produce independent red errors.

They should appear only as grouped attempt details, and only the aggregated all-failed result
should become an `ERROR`.

### Autonomous review

This policy makes the log severity map much more truthful, which is the core UX problem raised
by the user.

## Section 5: Implementation Shape

### Recommended code shape

Keep the implementation local to `src/crawl.ts`:

- add a small formatter/helper for severity-prefixed messages
- extend `PendingUrl` with guess-group context
- add a guess-group state map keyed by original URL
- funnel candidate result handling through a grouped finalizer
- update existing ad hoc `console.log` and `console.error` calls to the new formatter

Possible helper types:

- `LogLevel`
- `GuessAttemptKind = "original" | "guess"`
- `GuessAttemptOutcome`
- `GuessGroupState`

No queue-manager redesign is needed. The queue can remain responsible only for concurrency.

### Why localizing in `crawl.ts` is preferred

- the logic that decides retry vs fail vs skip already lives there
- guess grouping is crawl-policy state, not queue state
- it minimizes blast radius

### Autonomous review

This matches the current ownership boundaries in the codebase and avoids pulling policy into
`QueueManager` or `fetch.ts`.

## Section 6: Testing Strategy

### Required tests

1. A grouped-success case:
   - original URL fails
   - one guess succeeds
   - no standalone error is emitted for failed candidates
   - exactly one grouped notice/info block is emitted

2. A grouped-all-fail case:
   - original and all guesses fail
   - exactly one grouped error block is emitted
   - candidate detail lines appear together in that block

3. Retry / rate-limit visibility:
   - rate-limited primary URL logs a warning with delay and action
   - abort after three consecutive 429 responses logs a terminal error

4. Regression protection for current behavior:
   - guessed candidates are still fetched only once
   - successful guessed markdown is still saved correctly

### Test style

Capture `console.log`, `console.warn`, and `console.error` during crawl tests and assert on:

- call count
- severity prefix
- grouped message content

Prefer assertions on stable substrings instead of the entire formatted message so future wording
changes do not make the tests brittle.

### Autonomous review

The test plan is sufficient to protect the user-visible behavior without overspecifying every
string literal.

## Non-Goals

- introducing an external logger package
- adding JSON log output
- building a generic tracing system
- changing retry semantics beyond what is needed to log them clearly
- adding CLI flags for log verbosity in this pass

- The plan increases traceability for blocked or red pipelines.
- The implementation target stays local to `src/crawl.ts`, with no unnecessary architectural spread.
