# Parallel Crawl with Per-Domain Concurrency

## Problem

The crawler fetches pages one at a time sequentially. With hundreds of pages across multiple domains, HTTP round-trip latency (200-500ms each) makes the crawl take too long — 26+ minutes for the full set of seeds, vs ~3 minutes when there were fewer seeds.

## Design

### Architecture

Introduce a **centralized QueueManager** with per-domain concurrency control. All seed groups share the same QueueManager instance. Each domain gets its own internal queue with a semaphore limiting concurrent fetches.

```
all groups start in parallel (Promise.all)
  → seeds discover URLs
  → URLs submitted to QueueManager.fetch(url)
  → QueueManager routes to per-domain queue (by host+subdomain)
  → each domain queue runs up to 10 concurrent HTTP fetches
  → results returned to the calling group
```

### Components

#### QueueManager

- Single instance, shared across all groups.
- One public method: `fetch(url): Promise<FetchResult>`.
- Internally maintains a `Map<domain, DomainQueue>`, created lazily.
- Constructor accepts optional `concurrencyPerDomain` (default 10).

#### DomainQueue

- Internal to QueueManager.
- Tracks `inflight` count against `maxConcurrency`.
- Maintains a `pending` list of `{ url, resolve }` entries waiting for a slot.
- `pause(ms)` — stops launching new fetches for the given duration (triggered by 429).
- `drain()` — called after a fetch completes or pause expires; launches the next pending fetch if a slot is available.

#### CrawlGroup (refactored crawlGroup)

- Owns dedup state (`queued`, `fetched`, `failed` sets), scope rules, items, and urlResolution.
- Does NOT manage concurrency — just calls `queueManager.fetch(url)` for every discovered URL without awaiting individually.
- Reacts to resolved promises: saves content, parses new URLs, enqueues them.
- Waits for all in-flight fetches to complete before returning.

#### fetchWithRedirects (unchanged)

- Called by DomainQueue workers. No changes needed.

### Rate Limiting (429)

1. DomainQueue receives 429 from fetchWithRedirects.
2. Pauses the domain for `retryAfter` ms (or 5s default).
3. Returns `FetchResult` with `type: "rate-limited"` to the group.
4. Group does existing logic: increments `consecutive429s`, requeues URL, calls `queueManager.fetch(url)` again.
5. That new call waits in `DomainQueue.pending` until pause expires.
6. In-flight requests already sent continue — only new launches are held.
7. Abort-after-3-consecutive-429s logic remains in the group, unchanged.

### Concurrency Config

- Default: 10 concurrent fetches per domain.
- Overridable via CLI: `--concurrency 5`.
- Hardcoded default in QueueManager, no env var.

### Trade-offs

- **Unbounded pending promises**: When a seed discovers many URLs at once, they all become pending promises in DomainQueue. These are lightweight (URL + resolve callback), not HTTP connections. Negligible memory at our scale (hundreds to low thousands of URLs).
- **No backpressure on the group**: The group fires fetch calls as fast as it discovers URLs. Fine for our scale; would need a worker pool model for millions of URLs.

## Testing

### New tests

1. **DomainQueue**: submit N requests, verify only `maxConcurrency` in-flight at once; verify `pause(ms)` holds new fetches until elapsed.
2. **QueueManager**: submit URLs to different domains, verify routing to separate queues; verify per-domain independence (one domain paused, other keeps going).

### Existing tests

- `fetchWithRedirects`, `saveContent`, `buildMetadata`, `markRemovedItems`, `parseUrls` — unchanged.
- Integration test (local HTTP server with 4 pages) — should pass as-is since concurrency is transparent to group logic. Minor adjustment may be needed if `crawlGroup` signature changes.
